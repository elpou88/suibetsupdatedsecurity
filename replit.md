# SuiBets Platform - Crypto Sports Betting Platform

## Overview
SuiBets is a crypto sports betting platform built on the Sui blockchain, offering real-time betting across 30+ sports. It integrates multiple sports APIs for live scores and automated event tracking, utilizing blockchain for secure transactions and PostgreSQL for data persistence. The platform aims to provide a comprehensive and robust betting experience with a focus on real-time odds, secure on-chain betting, and a user-friendly interface, with the ambition to be a leading platform in the crypto sports betting market.

## System Architecture

### Frontend
- **Frameworks**: React 18 (TypeScript), Vite
- **Styling & UI**: Tailwind CSS, Framer Motion for animations, Radix UI for components
- **Data & Routing**: TanStack Query for data fetching, Wouter for routing
- **UI/UX Decisions**: Redesigned event cards with inline odds buttons, collapsible league sections, major leagues prioritized and expanded by default, quick bet functionality.

### Backend
- **Framework**: Express.js (TypeScript)
- **API**: RESTful design
- **Real-time**: WebSocket for live score updates
- **Data Aggregation**: Multi-API with resilience and fallback mechanisms
- **Authentication**: Session-based with optional blockchain authentication
- **Security**: Server-authoritative betting cutoff, rejection of stale event data, anti-exploit protections (rate limiting, cooldowns, max bets per event, event validation, settlement blocking). **Live odds staleness protection**: odds cache tracks score context (`cachedScore`); live odds have 3-minute TTL (`liveOddsCacheTTL`); both oracle sign and bet placement endpoints detect score changes and reject bets with stale odds (fail-closed — no cached odds = rejection). Parlay oracle signing validates each leg individually (existence + start-time checks) using `extractParlayLegIds()` parser that handles underscored event IDs. Admin auth is token-only (Bearer token via `Authorization` header) — no passwords in request bodies. Rate limiting: 100 req/min global on `/api/`, 20 req/min on `/api/admin/`. Body size limit: 50kb. CORS uses strict origin equality. Admin API response bodies are never logged. All admin wallet addresses use env vars (`ADMIN_WALLET_ADDRESS`, `VITE_ADMIN_WALLET_ADDRESS`). Oracle signatures bind wallet address + prediction to prevent replay attacks. Treasury status endpoint returns minimal data for non-admins. Parlay endpoints (both `/api/parlays` and `/api/bets/parlay`) enforce rate limits and cooldowns. Move contract uses u128 safe math for payout calculations with explicit u64 overflow assertion. Oracle sign-bet endpoint validates event existence and match status before signing, and enforces strict 32-byte Sui address format and 1000x max odds cap. Move void functions assert treasury balance before refund (no silent failures). Withdraw treasury asserts remaining balance covers outstanding liability. Deposit endpoint requires on-chain tx verification + sender address match (no graceful degradation bypass). Bet settlement and referral award endpoints require admin auth. SBETS off-chain bets require on-chain balance verification (single bets + parlays). **Security hardening (March 2026)**: Removed `isOnChainConfirmedEarly` bypass — ALL bets now go through odds verification and event status checks regardless of txHash/onChainBetId presence; no API keys hardcoded in service files (all read from env vars only); settlement worker uses single-flight mutex (`_settlementInProgress`) to prevent concurrent settlement cycles from overlapping; `determineBetOutcome()` now handles BTTS, Double Chance, and Half-Time markets correctly. Free bet failure is fail-closed (falls through to SBETS balance check). Referral award uses check-then-mutate pattern: eligibility checks (blocklist + on-chain ≥0.01 SUI) run BEFORE status update; both `/api/referral/award` and auto-award path in `/api/bets` use the same guards. XSS sanitized in streaming routes (escHtml + allowlist). SSRF blocked in streaming JS proxy (filename allowlist). DB-backed txHash dedup for social predictions/challenges (atomic INSERT with RETURNING, survives restarts). Unique constraints on social_prediction_bets.tx_id and social_challenge_participants.tx_hash. No 'user1' defaults in any financial path — all userId fields require real values. Balance/transactions/notifications endpoints require userId parameter (no data leaks). Cash-out and withdrawal endpoints verify wallet ownership. Settlement worker refuses to credit winnings to bets without userId. Currency resolution: `betCurrency = currency || feeCurrency || 'SUI'` (currency is optional with no default in PlaceBetSchema to avoid schema default overriding explicit feeCurrency). Event bet limit uses `external_event_id` fallback for non-numeric event IDs. Rate limit DB query uses `.toISOString()` for Date parameters (Postgres driver requires string, not Date object).

### Key Features
- **Sports Data Integration**: Aggregates real-time scores and odds from multiple providers.
- **Blockchain Integration**: Sui blockchain for secure transactions, SBETS token support, and Move smart contracts for betting and automated payouts.
- **Betting System**: Real-time odds, multiple market types, live betting via WebSockets, betting slip management, automated on-chain payouts, multi-leg parlay betting, gift bets (send bet winnings to another wallet). Esports betting (LoL + Dota 2, sportId 9).
- **Event Time Filtering**: All three event paths (esports fast path, free sports fast path, main upcoming path) strictly reject events with missing/invalid/past startTime. Dota 2 generates matchups from active pro leagues (never shows historical results as upcoming). LoL skips events without valid future startTime. All events require `startTime > now` and `!isNaN(startMs)` to appear in upcoming.
- **User Management**: Wallet-based authentication, user profiles, SUI and SBETS token balance management, user betting limits, referral system.
- **On-Chain Fund Flow**: Full on-chain dual-token system for bets and settlements via smart contracts, transparent treasury management, and fee accrual. Treasury Guard (`server/services/treasuryGuardService.ts`) wraps ALL outbound fund transfers (sendSuiToUser, sendSbetsToUser, withdrawFeesOnChain, withdrawFeesSbetsOnChain, withdrawTreasurySbetsOnChain) with per-transaction limits (50 SUI / 500K SBETS), daily caps (500 SUI / 5M SBETS), hourly tx count limit (100), large payout delay (30s for >20 SUI / >200K SBETS), emergency freeze capability, and persistent audit log table (`treasury_audit_log`). Auto-freezes on limit breach. Admin endpoints: GET `/api/admin/treasury-guard`, POST `/api/admin/treasury-guard/freeze`, POST `/api/admin/treasury-guard/unfreeze`.
- **Liability Tracking**: Explicit currency tracking, maximum stake limits, treasury pre-checks, on-chain bet synchronization.
- **Revenue Sharing (4-way 25% split)**: Platform revenue split into 4 equal parts — 25% SBETS holders, 25% Cetus LP providers, 25% treasury buffer, 25% platform profit. Holder claims: proportional to SBETS token holdings vs circulating supply. LP claims: proportional to liquidity share in Cetus SBETS-SUI pool (`0xa809b51e...`). Both claim types are weekly with per-wallet rate limits. Service: `server/services/cetusLpService.ts` (fetches LP positions from on-chain pool, handles wrapped/burned positions). Revenue distribution service: `server/services/revenueDistributionService.ts` (auto-withdraws 75% from on-chain treasury every 15min — holders 25% + profit 25% + LP 25%). Endpoints: `/api/revenue/stats`, `/api/revenue/claimable/:wallet`, `/api/revenue/claim`, `/api/revenue/lp-claimable/:wallet`, `/api/revenue/lp-claim`, `/api/revenue/lp-positions`. Storage: `platform_revenue_holders`, `platform_revenue_lp`, `platform_treasury_buffer`, `platform_profit` accounts. DB: `revenue_claims` table with `claim_type` column (`holder` or `lp`).
- **Social Network Effect Engine ("Predict Anything")**: Standalone /network page with custom prediction markets, viral challenges, public profiles, live chat, follow system, and leaderboard integration. Features on-chain prediction bets and challenge stakes, atomic pool updates, and automated resolution/settlement with anti-exploit security.
- **Live Streaming Section**: Proxies `streamed.pk` API for live and upcoming football matches with embedded playback.
- **zkLogin (Google OAuth)**: Full Sui zkLogin implementation for seedless wallet login via Google, integrated with on-chain betting.
- **Walrus Decentralized Storage**: Stores bet receipts on Walrus Protocol (mainnet) with multi-publisher fallback (publisher.walrus-mainnet.walrus.space, walrus-publisher.nodes.guru, publisher.walrus.space). Service: `server/services/walrusStorageService.ts`. Aggregators: aggregator.walrus-mainnet.walrus.space, aggregator.walrus.space. Receipt format v2.0 includes SuiBets branding (colors, logo, website links), bet details, blockchain info, and SHA-256 verification hash. Each bet receipt gets a Walrus blob ID stored in `bets.walrus_blob_id`. Receipt JSON cached in `bets.walrus_receipt_data`. Frontend shows "Verify on Walrus" link. Receipt viewer at `/api/walrus/receipt/:blobId` serves branded HTML page (XSS-safe) or JSON with `?json=true`. DNS for Walrus publishers doesn't resolve from Replit — works on Railway (production). Non-blocking: if all publishers fail, bet uses `local_<hash>` fallback.
- **SuiNS Integration**: Resolves wallet addresses to `.sui` domain names for enhanced UI.

### Architecture Model
- **Full On-Chain Model**: Bets placed directly on smart contracts, tracked in PostgreSQL for UI, settlements automated on-chain.
- **Capability-Based Security**: Smart contracts use AdminCap and OracleCap for access control.

### Anti-Exploit Odds Verification (FAIL-CLOSED)
- **`lookupServerOdds()`**: Verifies submitted odds against API-Sports/free sports/esports caches with 15% tolerance (`ODDS_TOLERANCE = 0.15`). Applied to `/api/bets`, `/api/oracle/sign-bet`, `/api/bets/parlay`, `/api/parlays`, `/api/on-chain-parlay`.
- **Fail-closed policy**: If event is found but selection can't be mapped → reject (`ODDS_UNVERIFIABLE`). If no reference odds at all → conservative 10x cap. Oracle signing uses 1000 bps conservative cap.
- **Max payout caps at bet placement**: 50 SUI / 25M SBETS (`MAX_PAYOUT_SUI`, `MAX_PAYOUT_SBETS` constants). Max stake: 100 SUI / 1M SBETS.
- **Max odds**: Zod schema caps at 50x (`validation.ts`).
- **On-chain parlay validation**: Strict schema enforcement — legs 2-10, numeric odds 1.01-50, totalOdds consistency check (5% tolerance vs computed product of leg odds).
